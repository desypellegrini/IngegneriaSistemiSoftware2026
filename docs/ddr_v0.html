Differential Drive Robot (DDR)

Introduction

Costruire un sistema software che comanda un Differential Drive Robot (DDR).

Requirements

Costruire un sistema software che comanda un Differential Drive Robot (DDR) in modo che, partendo dalla posizione iniziale HOME, il robot si sposti lungo il perimetro di una stanza rettangolare vuota.
Il DDR si deve fermare quando tornato in HOME.

Il DDR è anche sensibile ai dati rilevati da un Radar.
Quando il Radar rileva un 'intruso' a una distanza minore di una distanza prefissata ``DMIN``:
  • se il DDR si sta muovendo, il DDR si ferma fino a che il Radar non rileva più un 'intruso' così vicino
  • se il DDR è fermo in HOME, il DDR ruota su sè stesso fino a che un 'intruso' così vicino scompare

Requirement analysis

Il sistema software è composto dalle seguenti entità e proprietà:

  • la Stanza e HOME
    Per formalizzare la stanza, la modello come un rettangolo in un piano euclideo di dimensioni B (base) e H (altezza). Tuttavia poiché il robot non è un punto ma occupa un'area fisica (che assumiamo quadrata di lato R), è utile rappresentare la stanza come una griglia di celle. Introduco quindi l'unità di misura "unità robotica" (corrispondente al lato R), che rende il modello indipendente dalle unità fisiche e più vicino alla logica di movimento del software.
    • Stanza: formalizzata tramite le variabili B (numero di celle su x), H (numero di celle su y) e R (dimensione della cella in metri).
    • HOME: rappresenta la cella di partenza e arrivo del percorso. Viene modellata come una cella specifica (variabile di stato con coordinate x, y) all'interno della griglia.
    Si formalizza la struttura con il seguente codice Kotlin: 
    data class Stanza(val B: Int, val H: Int, val R: Double) {
    // Posizione HOME: cella (0, 0)
    val home: Posizione = Posizione(0, 0)
    
    // Dimensioni fisiche in metri  
    val larghezzaMetri: Double get() = B * R
    val altezzaMetri: Double get() = H * R
}

data class Posizione(val x: Int, val y: Int)
  • Il Radar
    Il Radar è un'entità di input (che modello come attore) incaricata di monitorare l'ambiente e rilevare un intruso a una distanza D. La sua proprietà fondamentale è rendere disponibile questa informazione al sistema. In fase di analisi dei requisiti, formalizziamo due possibili modelli di interazione:
    1. modello a eventi: il Radar agisce in modo autonomo, emettendo periodicamente un segnale asincrono. Questa soluzione è indicata se non conosciamo a priori chi utilizzerà l'informazione. 
      // MODELLO ANALISTA (EVENT)
Event sonardata : distance(D)
QActor radar context ctxddr {
    State s0 initial { }
    Goto rileva
    State rileva {
        // rileva distanza D
        emit sonardata : distance(D)
        delay 1000
    }
    Goto rileva
}
    2. modello a richiesta: il Radar comunica la distanza solo in risposta a una sollecitazione esplicita del sistema. Questo garantisce che il dato venga scambiato solo quando necessario. 
      // MODELLO ANALISTA (REQUEST)
Request getDistance : getDistance(D)
Reply distance : distance(D) for getDistance
QActor radar context ctxddr {
    State s0 initial { }
    Transition t0 whenRequest getDistance -> handleRequest
    State handleRequest {
        replyTo getDistance with distance : distance(D)
    }
    Goto s0
}
  • DDR Robot e Mind
    Dall'analisi emerge una distinzione fondamentale tra il corpo e la mente del sistema:
    • DDR Robot (corpo): è un'entità puramente reattiva e priva di autonomia decisionale. Rappresenta l'attuatore di comandi fornito dal committente (già esistente come codice). Le sue capacità sono limitate all'esecuzione di movimenti elementari (avanti, ruota, stop) su richiesta di entità esterne.
    • Mind (mente): è il componente software proattivo che implementa la business logic. È l'unico attore del sistema che conosce l'obiettivo (percorrere il perimetro) e deve coordinare il DDR in base agli stimoli del Radar.
Per formalizzare il comportamento della Mind e la sua reattività agli eventi ambientali, si delinea il seguente modello dell'analista: 
// MODELLO ANALISTA PER LA BUSINESS LOGIC
QActor mind context ctxddr {
    [# var DMIN = 0.5 #] // soglia definita nei requisiti
    
    State s0 initial {
        // logica proattiva: comando del DDR per il perimetro
    }
    Transition t0 
        whenEvent sonardata -> handleRadar
        
    State handleRadar {
        onMsg( sonardata : distance(D) ) {
            // Se D < DMIN:
            // 1. Stop se il DDR è in movimento lungo il perimetro
            // 2. Rotazione se il DDR è fermo in HOME
            // 3. Attesa fino a D > DMIN per il ripristino
        }
    }
    Goto s0 // Ritorno allo stato interrotto
}
