fireflysonar

Introduction

Costruire il simulatore di un sistema costituito da 3 lucciole.
Questo progetto è l'evoluzione del sistema fireflysynch.

Requirements

Inizialmente, le lucciole lampeggiamo ciascuna con una frequenza (costante) propria.
    B. Il sistema è dotato anche di un sonar. Quando il sonar rileva un ostacolo a una distanza minore di una distanza prefissata DMIN, le lucciole devoono lampeggiare in modo sincronizzato. Quando la distanza dell’ostacolo (ri)supera DMIN, le lucciole tornano a lampeggiare in modo random.

Requirement analysis

Il sistema software è composto dalle seguenti entità e proprietà:
  • Lucciola: entità capace di lampeggiare ciclicamente (accensione/spegnimento). Inizialmente opera con un ritmo proattivo autonomo caratterizzato da una frequenza costante propria. Viene modellata come un Actor in quanto possiede un comportamento interno e autonomo (brillamento).
  • Sonar: dispositivo di input incaricato di rilevare la distanza D di un ostacolo. L'informazione deve essere resa disponibile al sistema per coordinare il comportamento delle lucciole.
    Esistono due modelli formali per questa interazione (prospettiva dell'analista):
    1. modello a eventi: il Sonar emette periodicamente un segnale asincrono. 
      Event sonardata : sonar(D)
    2. modello a richiesta: il sistema interroga il Sonar quando necessario. 
      Request getDistance : getDistance(D)
  • DMIN: valore soglia prefissato che determina il passaggio tra il comportamento proattivo (random) e quello reattivo (sincronizzato).
  • Stati del sistema:
    • Modalità Random: ogni lucciola opera indipendentemente con il proprio timer locale.
    • Modalità Sincronizzata: stato attivato quando D < DMIN, in cui tutte le lucciole adottano un periodo di lampeggio uniforme.
Sulla base di quanto analizzato sopra, si formalizza la struttura del sistema con il seguente modello QAK iniziale: 
System fireflysonar

// messaggi e segnali identificati dal dominio
Dispatch cellstate : cellstate(X,Y,COLOR)
Event sync : sync(T)          // allarme che sincronizza sulla frequenza T
Event desync : desync(N)      // torna random
Event sonardata : sonar(D)    // dati simulati dal sonar

Context ctxfirefly ip [ host="localhost" port=8040 ] 

// La Lucciola è un Actor con comportamento proprio
QActor firefly context ctxfirefly { 
    State s0 initial { } 
}

// Il Sonar è l'entità di input ambientale
QActor sonarmock context ctxfirefly { 
    State s0 initial { } 
}

Problem analysis

L'attore sonarmock emette periodicamente eventi sonardata per riflettere lo stato dell'ambiente.
Dall'analisi dei requisiti emerge che Sonar e Lucciola sono entità disaccoppiate. Scelgo di introdurre un Controller dedicato che agisce come mente del sistema. In questo modo, il sonarmock produce dati, mentre il controller gestisce la business logic (confronto con DMIN ed emissione dei comandi di stato).

Sono state valutate diverse opzioni per imporre il ritmo comune:
  1. Uso di un orchestratore che impone una nuova frequenza tramite messaggi diretti (meno scalabile).
  2. Sincronizzazione distribuita basata sull'osservazione (coreografia, più complessa).
  3. Emissione di eventi broadcast (scelta): l'uso di emit sync/desync permette a un numero arbitrario di lucciole di percepire il cambio di stato senza che il Controller debba conoscerne l'identità.

Ogni lucciola memorizza la propria frequenza random in una variabile locale MyFreq durante lo stato iniziale s0. Questo permette di ripristinare il comportamento proattivo originale dopo la ricezione dell'evento desync.
La lucciola deve reagire al sonar anche mentre sta aspettando il momento di lampeggiare. L'uso della primitiva whenTimeVar Timer combinata con whenEvent sync/desync permette di interrompere immediatamente il delay al variare delle condizioni ambientali.

Test plans

Project

Il progetto realizza un'architettura distribuita composta da tre tipologie di attori, ognuno con una responsabilità definita:
    • ctxfirefly: nodo principale che ospita la logica di simulazione (porta 8040).
    • ctxgrid: nodo esterno per la visualizzazione dello stato delle celle (porta 8050).
  • Il sonarmock emette dati; il controller funge da orchestratore, trasformando le letture di distanza in comandi di sistema (sync/desync) basati sulla soglia DMIN.
  • Per quanto riguarda l'attore Firefly, la transizione principale utilizza whenTimeVar Timer per il lampeggio proattivo, ma rimane in ascolto degli eventi globali emessi dal controller per variare il proprio parametro temporale.
  • Il codice completo della logica sonar è disponibile nel file fireflysonar.qak.
